# Cookie Pipeline 本地主入口

主入口文件：

```text
sites/waimai_guide_ele_me/src/cookie_pipeline.py
```

这个入口的设计目标是：你把 Cookie 放到本地文件里，程序自动完成离线诊断链路，包括 Cookie 解析、`_m_h5_tk` token 提取、过期时间判断、mtop `data` 构造、`sign` 计算、`bx-umidtoken` 形态检查、安全参数槽位检查和脱敏报告输出。

它不会发送目标业务请求，也不会内置生产安全参数绕过逻辑。

## 1. 准备 Cookie 文件

建议把 Cookie 放到一个临时本地文件，例如：

```text
runtime/eleme_cookie.txt
```

文件内容就是浏览器里复制出来的一整条 Cookie header，例如：

```text
ubt_ssid=...; cna=...; _m_h5_tk=..._...; _m_h5_tk_enc=...; xqkp=...=; x5sec=...; tfstk=...
```

注意：

- 不要把真实 Cookie 提交到 git。
- 不要把 Cookie 原值贴到报告或文档里。
- 跑完测试后可以删除这个临时文件。

## 2. 运行命令

基础命令：

```powershell
.\venv\Scripts\python.exe .\sites\waimai_guide_ele_me\src\cookie_pipeline.py --cookie-file .\runtime\eleme_cookie.txt
```

打印脱敏报告到屏幕：

```powershell
.\venv\Scripts\python.exe .\sites\waimai_guide_ele_me\src\cookie_pipeline.py --cookie-file .\runtime\eleme_cookie.txt --print
```

指定关键词、页码、坐标：

```powershell
.\venv\Scripts\python.exe .\sites\waimai_guide_ele_me\src\cookie_pipeline.py `
  --cookie-file .\runtime\eleme_cookie.txt `
  --keyword "盖码饭" `
  --latitude "28.231626" `
  --longitude "112.884453" `
  --page 2 `
  --limit 5
```

默认报告输出到：

```text
sites/waimai_guide_ele_me/docs/cookie-pipeline-report.json
```

## 3. 自动完成的离线链路

脚本会自动做这些事：

| 步骤 | 说明 |
| --- | --- |
| 读取 Cookie | 支持 `--cookie-file`、`--cookie`、`ELEME_COOKIE` 环境变量 |
| 解析 Cookie | 输出 key 列表和关键 Cookie 的存在状态/长度 |
| 提取 token | 从 `_m_h5_tk` 的 `_` 前面提取 mtop token |
| 判断时效 | 读取 `_m_h5_tk` 后缀时间戳，计算是否已过期 |
| 解析 `x5sec` | 尝试 hex 解码并读取 `t` 字段，只输出形态信息 |
| 构造 body | 生成 `data={outer-json}` 的 form body |
| 计算 `sign` | 使用 `md5(token + "&" + t + "&" + appKey + "&" + data)` |
| 构造 ET 输入材料 | 生成用于 `etSign(url)` 的 URL-like 材料摘要 |
| 检查 `bx-umidtoken` | 从 `xqkp` 去掉末尾 `=` 得到 header 形态 |
| 输出报告 | 只输出长度、存在状态、短摘要和下一步提示 |

## 4. 输出怎么看

重点看这些字段：

```json
{
  "readiness": {
    "canComputeMtopSign": true,
    "hasCookiePrereqs": true,
    "mtopTokenExpiredNow": false,
    "needsAuthorizedRuntimeForSecurity": true,
    "willSendRequest": false
  }
}
```

含义：

| 字段 | 说明 |
| --- | --- |
| `canComputeMtopSign` | `_m_h5_tk` token、`t`、`appKey`、`data` 是否足够计算 `sign` |
| `hasCookiePrereqs` | `_m_h5_tk`、`_m_h5_tk_enc`、`xqkp`、`cna` 是否齐全 |
| `mtopTokenExpiredNow` | 当前本地时间看 `_m_h5_tk` 是否已过期 |
| `needsAuthorizedRuntimeForSecurity` | 是否还缺同浏览器运行时生成的 `bx_et/x-ele-check` |
| `willSendRequest` | 固定为 `false`，脚本不会请求业务接口 |

`request.query.sign` 只输出长度和摘要：

```json
"sign": {
  "present": true,
  "length": 32,
  "sha256_16": "..."
}
```

`cookies.required` 会告诉你关键 Cookie 是否齐全：

```json
"_m_h5_tk": {"present": true, "length": 46},
"_m_h5_tk_enc": {"present": true, "length": 32},
"x5sec": {"present": true, "length": 300},
"xqkp": {"present": true, "length": 68},
"tfstk": {"present": true, "length": 469},
"cna": {"present": true, "length": 24}
```

## 5. 安全参数槽位

当前入口不会内置 `bx_et` 和 `x-ele-check` 的生产生成逻辑。报告里会显示：

```json
"provider": {
  "available": false,
  "reason": "No authorized provider configured. bx_et and x-ele-check are not generated by this entrypoint.",
  "providerContract": [
    "etSign(url, ctx)",
    "xCheck(prefix, api, ctx)",
    "xEleUa(ctx)",
    "umidToken(ctx)"
  ]
}
```

如果你在自己的授权环境里有合法 provider，可以通过：

```powershell
.\venv\Scripts\python.exe .\sites\waimai_guide_ele_me\src\cookie_pipeline.py `
  --cookie-file .\runtime\eleme_cookie.txt `
  --provider .\sites\waimai_guide_ele_me\src\my_authorized_provider.js
```

provider 返回值仍只会在报告里以长度和摘要展示。

## 6. 常见判断

| 现象 | 大概率原因 |
| --- | --- |
| `canComputeMtopSign=false` | 缺 `_m_h5_tk`、`t`、`appKey` 或 `data` |
| `mtopTokenExpiredNow=true` | `_m_h5_tk` 已过期，需要刷新浏览器 Cookie |
| `_m_h5_tk_enc` 缺失 | Cookie jar 不完整 |
| `xqkp` 缺失 | `bx-umidtoken` 无法从 Cookie 推导 |
| `x5sec.decodeOk=false` | `x5sec` 不是当前预期的 hex JSON 形态或已损坏 |
| `needsAuthorizedRuntimeForSecurity=true` | 还缺同浏览器状态下的 `bx_et/x-ele-check` |

## 7. 边界

这个入口只做本地检查和报告生成：

- 不请求 `waimai-guide.ele.me`。
- 不输出真实 Cookie 值。
- 不输出可直接复用的完整生产 curl。
- 不把不同会话的 Cookie、安全 header、`bx_et`、`x-ele-check` 混在一起。
